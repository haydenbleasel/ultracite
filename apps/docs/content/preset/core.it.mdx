La configurazione core di Ultracite fornisce un set completo di regole di linting e impostazioni di formattazione agnostiche al framework. Questa è la base che tutte le altre configurazioni specifiche per framework estendono.

## Installazione

La configurazione core è inclusa automaticamente quando esegui:

```bash
npx ultracite init
```

Oppure aggiungila manualmente al tuo `biome.jsonc`:

```jsonc title="biome.jsonc"
{
  "extends": ["ultracite/core"]
}
```

## Panoramica

La configurazione core include centinaia di regole di linting rigorose che coprono:

- **Accessibilità (a11y)**: Imporre attributi ARIA, HTML semantico e navigazione da tastiera
- **Complessità**: Ridurre la complessità cognitiva e applicare le best practice
- **Correttezza**: Prevenire errori comuni e garantire sicurezza dei tipi
- **Prestazioni**: Ottimizzare il codice per migliori prestazioni in esecuzione
- **Sicurezza**: Prevenire vulnerabilità di sicurezza
- **Stile**: Applicare uno stile di codice coerente
- **Sospetti**: Individuare pattern sospetti che potrebbero indicare bug

## Impostazioni del formattatore

- **Indentazione**: 2 spazi
- **Larghezza riga**: 80 caratteri
- **Fine linea**: LF (stile Unix)
- **Punto e virgola**: Sempre richiesti
- **Virgolette**: Virgolette doppie per JavaScript/TypeScript e JSON
- **Trailing Commas**: Stile ES5 (array/oggetti multilinea)
- **Parentesi per arrow**: Includere sempre le parentesi

## Caratteristiche principali

### Rigidità di TypeScript

La configurazione core abilita i controlli strict di TypeScript:

- Sconsiglia l'uso di `any` (`noExplicitAny: error`)
- Richiede la gestione di `null/undefined`
- Preferisce tipi espliciti in determinate situazioni
- Applica la sicurezza dei tipi attraverso il codebase

### Equivalenti raccomandati di ESLint

Regole comuni e best-practice incluse:

- No variabili non utilizzate (`noUnusedVariables: error`)
- No import non utilizzati (`noUnusedImports: error`)
- No eval esplicito (`noGlobalEval: error`)
- No prototype pollution (`noPrototypeBuiltins: error`)

### Accessibilità di default

Regole equivalenti agli standard di accessibilità:

- Gli attributi ARIA devono essere validi
- Preferire elementi HTML semantici
- Richiedere il supporto alla navigazione da tastiera
- Testo alternativo richiesto per le immagini

### Supporto Node.js

Gestione specifica per l'ambiente:

- Riconosce le globali di Node.js (`module`, `process`)
- Impone il protocollo `node:` per i moduli built-in
- Supporta sia CommonJS che ESM

## Note

Alcune regole sono disabilitate perché troppo lente da eseguire con lo Scanner. Come indicato nella [documentazione di Biome](https://biomejs.dev/guides/investigate-slowness/#first-things-first), le regole abilitate nel progetto sono note per causare overhead di prestazioni in cambio di analisi più avanzata.

Nei nostri test, abbiamo riscontrato che queste regole possono fare la differenza tra un lint di 300ms e uno di 13s.
