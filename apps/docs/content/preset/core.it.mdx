La configurazione core di Ultracite fornisce un insieme completo di regole di linting e impostazioni di formattazione indipendenti dal framework. Questa è la base che tutte le altre configurazioni specifiche per framework estendono.

## Installazione

La configurazione core è inclusa automaticamente quando esegui:

```bash
npx ultracite init
```

Oppure aggiungila manualmente al tuo `biome.jsonc`:

```jsonc title="biome.jsonc"
{
  "extends": ["ultracite/core"]
}
```

## Panoramica

La configurazione core include centinaia di regole di linting rigorose che coprono:

- **Accessibilità (a11y)**: Far rispettare gli attributi ARIA, l'HTML semantico e la navigazione tramite tastiera
- **Complessità**: Ridurre la complessità cognitiva e far rispettare le best practice
- **Correttezza**: Prevenire errori comuni e far rispettare la sicurezza dei tipi
- **Prestazioni**: Ottimizzare il codice per migliori prestazioni a runtime
- **Sicurezza**: Prevenire vulnerabilità di sicurezza
- **Stile**: Far rispettare uno stile di codice coerente
- **Sospetti**: Individuare pattern sospetti che potrebbero indicare bug

## Impostazioni del formatter

- **Indentazione**: 2 spazi
- **Lunghezza riga**: 80 caratteri
- **Fine riga**: LF (stile Unix)
- **Punto e virgola**: Sempre richiesti
- **Virgolette**: Virgolette doppie per JavaScript/TypeScript e JSON
- **Virgole finali**: Stile ES5 (array/oggetti multilinea)
- **Parentesi per arrow**: Includere sempre le parentesi

## Caratteristiche principali

### Rigore TypeScript

La configurazione core abilita i controlli rigorosi di TypeScript:

- Discourage l'uso di `any` (`noExplicitAny: error`)
- Richiede la gestione di `null/undefined`
- Preferisce i tipi espliciti in determinate situazioni
- Fa rispettare la sicurezza dei tipi in tutto il codebase

### Equivalenti consigliati da ESLint

Regole comuni di best-practice incluse:

- Nessuna variabile inutilizzata (`noUnusedVariables: error`)
- Nessun import inutilizzato (`noUnusedImports: error`)
- Nessun eval esplicito (`noGlobalEval: error`)
- Nessuna prototype pollution (`noPrototypeBuiltins: error`)

### Accessibilità di default

Regole equivalenti a standard di accessibilità:

- Gli attributi ARIA devono essere validi
- Sono preferiti elementi HTML semantici
- È richiesto il supporto alla navigazione tramite tastiera
- Testo alternativo per le immagini richiesto

### Supporto Node.js

Gestione specifica per l'ambiente:

- Riconosce le globali di Node.js (`module`, `process`)
- Fa rispettare il protocollo `node:` per i moduli built-in
- Supporta sia CommonJS che ESM

## Note

Alcune regole sono disabilitate perché sono troppo lente da eseguire con lo Scanner. Come indicato nella [documentazione di Biome](https://biomejs.dev/guides/investigate-slowness/#first-things-first), le regole abilitate nel progetto sono note per causare overhead di prestazioni in cambio di analisi più avanzate.

Nei nostri test, abbiamo riscontrato che queste regole possono fare la differenza tra un lint di 300 ms e uno di 13 s.
