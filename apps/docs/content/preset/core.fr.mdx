La configuration de base d'Ultracite fournit un ensemble complet de règles de linting et de paramètres de formatage indépendants du framework. C'est la fondation sur laquelle toutes les autres configurations spécifiques aux frameworks s'étendent.

## Installation

La configuration de base est incluse automatiquement lorsque vous exécutez :

```bash
npx ultracite init
```

Ou ajoutez-la manuellement à votre `biome.jsonc` :

```jsonc title="biome.jsonc"
{
  "extends": ["ultracite/core"]
}
```

## Aperçu

La configuration de base inclut des centaines de règles de linting strictes couvrant :

- **Accessibilité (a11y)** : Imposer les attributs ARIA, le HTML sémantique et la navigation au clavier
- **Complexité** : Réduire la complexité cognitive et appliquer les bonnes pratiques
- **Exactitude** : Prévenir les erreurs courantes et garantir la sûreté des types
- **Performance** : Optimiser le code pour de meilleures performances à l'exécution
- **Sécurité** : Prévenir les vulnérabilités de sécurité
- **Style** : Imposer un style de code cohérent
- **Suspicieux** : Repérer les motifs suspects susceptibles d'indiquer des bogues

## Paramètres du formatteur

- **Indentation** : 2 espaces
- **Largeur de ligne** : 80 caractères
- **Fin de ligne** : LF (style Unix)
- **Point-virgules** : Toujours requis
- **Guillemets** : Guillemets doubles pour JavaScript/TypeScript et JSON
- **Virgules finales** : Style ES5 (tableaux/objets multi-lignes)
- **Parenthèses des flèches** : Toujours inclure les parenthèses

## Principales fonctionnalités

### Rigueur TypeScript

La configuration de base active les vérifications strictes de TypeScript :

- Décourage l'utilisation de `any` (`noExplicitAny: error`)
- Exige la gestion de `null/undefined`
- Préfère des types explicites dans certaines situations
- Imposer la sûreté des types à travers la base de code

### Équivalents recommandés d'ESLint

Des règles de bonnes pratiques courantes sont incluses :

- Pas de variables inutilisées (`noUnusedVariables: error`)
- Pas d'importations inutilisées (`noUnusedImports: error`)
- Pas d'`eval` explicite (`noGlobalEval: error`)
- Pas de pollution de prototype (`noPrototypeBuiltins: error`)

### Accessibilité par défaut

Règles équivalentes aux standards d'accessibilité :

- Les attributs ARIA doivent être valides
- Privilégier les éléments HTML sémantiques
- Prise en charge de la navigation au clavier requise
- Texte alternatif pour les images requis

### Prise en charge de Node.js

Gestion spécifique à l'environnement :

- Reconnaît les globals Node.js (`module`, `process`)
- Imposer le protocole `node:` pour les modules intégrés
- Prend en charge à la fois CommonJS et ESM

## Remarques

Certaines règles sont désactivées parce qu'elles sont trop lentes à exécuter avec le Scanner. Comme indiqué dans la [documentation Biome](https://biomejs.dev/guides/investigate-slowness/#first-things-first), les règles de projet activées sont connues pour provoquer une surcharge de performance en échange d'analyses plus avancées.

Dans nos tests, nous avons constaté que ces règles peuvent faire la différence entre 300 ms et 13 s de lint.
