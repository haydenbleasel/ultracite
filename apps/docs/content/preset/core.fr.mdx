La configuration core d'Ultracite fournit un ensemble complet de règles de linting agnostiques au framework et des paramètres de formatage. C'est la base que toutes les autres configurations spécifiques aux frameworks étendent.

## Installation

La configuration core est incluse automatiquement lorsque vous exécutez :

```bash
npx ultracite init
```

Ou ajoutez-la manuellement à votre `biome.jsonc` :

```jsonc title="biome.jsonc"
{
  "extends": ["ultracite/core"]
}
```

## Aperçu

La configuration core inclut des centaines de règles de linting strictes couvrant :

- **Accessibilité (a11y)** : Faire respecter les attributs ARIA, le HTML sémantique et la navigation au clavier
- **Complexité** : Réduire la complexité cognitive et appliquer les bonnes pratiques
- **Correction** : Prévenir les erreurs courantes et garantir la sécurité des types
- **Performance** : Optimiser le code pour de meilleures performances d'exécution
- **Sécurité** : Prévenir les vulnérabilités de sécurité
- **Style** : Imposer un style de code cohérent
- **Suspicieux** : Détecter des motifs suspects pouvant indiquer des bugs

## Paramètres du formateur

- **Indentation** : 2 espaces
- **Largeur de ligne** : 80 caractères
- **Fin de ligne** : LF (style Unix)
- **Point-virgules** : Toujours requis
- **Guillemets** : Guillemets doubles pour JavaScript/TypeScript et JSON
- **Virgules finales** : Style ES5 (tableaux/objets multi-lignes)
- **Parenthèses des flèches** : Toujours inclure les parenthèses

## Principales fonctionnalités

### Rigueur TypeScript

La configuration core active les vérifications strictes de TypeScript :

- Décourage l'utilisation de `any` (`noExplicitAny: error`)
- Exige la gestion de `null/undefined`
- Préfère les types explicites dans certaines situations
- Applique la sécurité des types à travers la base de code

### Équivalents recommandés d'ESLint

Les règles courantes de bonnes pratiques sont incluses :

- Pas de variables inutilisées (`noUnusedVariables: error`)
- Pas d'importations inutilisées (`noUnusedImports: error`)
- Pas d'utilisation explicite d'eval (`noGlobalEval: error`)
- Pas de prototype pollution (`noPrototypeBuiltins: error`)

### Accessibilité par défaut

Règles équivalentes aux standards d'accessibilité :

- Les attributs ARIA doivent être valides
- Les éléments HTML sémantiques sont préférés
- Le support de navigation au clavier est requis
- Texte alternatif pour les images requis

### Prise en charge de Node.js

Gestion spécifique à l'environnement :

- Reconnaît les globals Node.js (`module`, `process`)
- Imposer le protocole `node:` pour les modules intégrés
- Prend en charge à la fois CommonJS et ESM

## Notes

Certaines règles sont désactivées car elles sont trop lentes à exécuter avec le Scanner. Comme indiqué dans les [docs Biome](https://biomejs.dev/guides/investigate-slowness/#first-things-first), certaines règles de projet activées sont connues pour entraîner une surcharge de performance en échange d'une analyse plus avancée.

Dans nos tests, nous avons constaté que ces règles peuvent faire la différence entre 300ms et 13s de lint.
