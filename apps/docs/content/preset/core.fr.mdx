La configuration de base d'Ultracite fournit un ensemble complet de règles de linting indépendantes du framework et de paramètres de formatage. C'est la fondation que toutes les autres configurations spécifiques aux frameworks étendent.

## Installation

La configuration de base est incluse automatiquement lorsque vous exécutez :

```bash
npx ultracite init
```

Ou ajoutez-la manuellement à votre `biome.jsonc` :

```jsonc title="biome.jsonc"
{
  "extends": ["ultracite/core"]
}
```

## Aperçu

La configuration de base inclut des centaines de règles de lint strictes couvrant :

- **Accessibilité (a11y)** : Faire respecter les attributs ARIA, le HTML sémantique et la navigation au clavier
- **Complexité** : Réduire la complexité cognitive et appliquer les meilleures pratiques
- **Correctness** : Prévenir les erreurs courantes et assurer la sûreté des types
- **Performance** : Optimiser le code pour de meilleures performances à l'exécution
- **Sécurité** : Prévenir les vulnérabilités de sécurité
- **Style** : Imposer un style de code cohérent
- **Suspicious** : Détecter les motifs suspects pouvant indiquer des bugs

## Paramètres du formateur

- **Indentation** : 2 espaces
- **Largeur de ligne** : 80 caractères
- **Fin de ligne** : LF (style Unix)
- **Point-virgules** : Toujours requis
- **Guillemets** : Guillemets doubles pour JavaScript/TypeScript et JSON
- **Virgules finales** : Style ES5 (tableaux/objets multi-lignes)
- **Parenthèses fléchées** : Toujours inclure les parenthèses

## Principales fonctionnalités

### Rigueur TypeScript

La configuration de base active les vérifications strictes de TypeScript :

- Décourage l'utilisation de `any` (`noExplicitAny: error`)
- Exige la gestion de `null/undefined`
- Préfère des types explicites dans certaines situations
- Impose la sûreté des types dans l'ensemble du codebase

### Équivalents recommandés d'ESLint

Les règles courantes de bonnes pratiques sont incluses :

- Pas de variables inutilisées (`noUnusedVariables: error`)
- Pas d'importations inutilisées (`noUnusedImports: error`)
- Pas d'usage explicite d'eval (`noGlobalEval: error`)
- Pas de prototype pollution (`noPrototypeBuiltins: error`)

### Accessibilité par défaut

Règles équivalentes aux standards d'accessibilité :

- Les attributs ARIA doivent être valides
- Les éléments HTML sémantiques sont préférés
- Le support de la navigation au clavier est requis
- Texte alternatif pour les images requis

### Prise en charge de Node.js

Gestion spécifique à l'environnement :

- Reconnaît les globaux Node.js (`module`, `process`)
- Implique le protocole `node:` pour les modules intégrés
- Supporte à la fois CommonJS et ESM

## Notes

Certaines règles sont désactivées car elles sont trop lentes à exécuter avec le Scanner. Comme indiqué dans la [documentation de Biome](https://biomejs.dev/guides/investigate-slowness/#first-things-first), les règles de projet activées sont connues pour causer une surcharge de performance en échange d'analyses plus avancées.

Dans nos tests, nous avons constaté que ces règles peuvent faire la différence entre un lint de 300 ms et un lint de 13 s.
