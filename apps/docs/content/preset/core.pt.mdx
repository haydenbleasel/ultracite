A configuração core do Ultracite fornece um conjunto abrangente de regras de linting e definições de formatação independentes de framework. Esta é a base que todas as outras configurações específicas de framework estendem.

## Instalação

A configuração core é incluída automaticamente quando você executa:

```bash
npx ultracite init
```

Ou adicione-a manualmente ao seu `biome.jsonc`:

```jsonc title="biome.jsonc"
{
  "extends": ["ultracite/core"]
}
```

## Visão geral

A configuração core inclui centenas de regras de linting rigorosas cobrindo:

- **Acessibilidade (a11y)**: Aplicar atributos ARIA, HTML semântico e navegação por teclado
- **Complexidade**: Reduzir complexidade cognitiva e aplicar melhores práticas
- **Corretude**: Prevenir erros comuns e aplicar segurança de tipos
- **Performance**: Otimizar o código para melhor desempenho em tempo de execução
- **Segurança**: Prevenir vulnerabilidades de segurança
- **Estilo**: Aplicar estilo de código consistente
- **Suspeitas**: Capturar padrões suspeitos que podem indicar bugs

## Configurações do formatador

- **Indentação**: 2 espaços
- **Largura da linha**: 80 caracteres
- **Final de linha**: LF (estilo Unix)
- **Ponto e vírgula**: Sempre obrigatório
- **Aspas**: Aspas duplas para JavaScript/TypeScript e JSON
- **Vírgulas finais**: Estilo ES5 (arrays/objetos multilinha)
- **Parênteses em arrow functions**: Incluir sempre parênteses

## Principais recursos

### Rigor do TypeScript

A configuração core ativa as checagens estritas do TypeScript:

- Desencoraja o uso de `any` (`noExplicitAny: error`)
- Requer tratamento de `null/undefined`
- Prefere tipos explícitos em certas situações
- Impõe segurança de tipos em toda a base de código

### Equivalentes recomendados do ESLint

Regras comuns de melhores práticas estão incluídas:

- Sem variáveis não usadas (`noUnusedVariables: error`)
- Sem imports não usados (`noUnusedImports: error`)
- Sem eval explícito (`noGlobalEval: error`)
- Sem prototype pollution (`noPrototypeBuiltins: error`)

### Acessibilidade por padrão

Regras equivalentes a padrões de acessibilidade:

- Atributos ARIA devem ser válidos
- Elementos HTML semânticos preferidos
- Suporte à navegação por teclado exigido
- Texto alternativo para imagens obrigatório

### Suporte a Node.js

Tratamento específico por ambiente:

- Reconhece globais do Node.js (`module`, `process`)
- Aplica o protocolo `node:` para módulos internos
- Suporta CommonJS e ESM

## Notas

Certas regras estão desativadas porque são muito lentas para rodar com o Scanner. Como observado na [documentação do Biome](https://biomejs.dev/guides/investigate-slowness/#first-things-first), rules ativadas no projeto são conhecidas por causar overhead de performance em troca de uma análise mais avançada.

Em nossos testes, descobrimos que essas regras podem fazer a diferença entre um lint de 300ms e um de 13s.
