A configuração núcleo do Ultracite fornece um conjunto abrangente de regras de linting e definições de formatação independentes de framework. Esta é a base que todas as outras configurações específicas de framework estendem.

## Instalação

A configuração núcleo é incluída automaticamente quando você executa:

```bash
npx ultracite init
```

Ou adicione-a manualmente ao seu `biome.jsonc`:

```jsonc title="biome.jsonc"
{
  "extends": ["ultracite/core"]
}
```

## Visão geral

A configuração núcleo inclui centenas de regras de linting rigorosas cobrindo:

- **Acessibilidade (a11y)**: Aplicar atributos ARIA, HTML semântico e navegação por teclado
- **Complexidade**: Reduzir a complexidade cognitiva e aplicar boas práticas
- **Correção**: Prevenir erros comuns e impor segurança de tipos
- **Performance**: Otimizar o código para melhor desempenho em tempo de execução
- **Segurança**: Prevenir vulnerabilidades de segurança
- **Estilo**: Aplicar um estilo de código consistente
- **Suspeitas**: Capturar padrões suspeitos que podem indicar bugs

## Configurações do formatador

- **Indentação**: 2 espaços
- **Largura de linha**: 80 caracteres
- **Fim de linha**: LF (estilo Unix)
- **Ponto e vírgula**: Sempre exigidos
- **Aspas**: Aspas duplas para JavaScript/TypeScript e JSON
- **Vírgulas finais**: Estilo ES5 (arrays/objetos multilinha)
- **Parênteses em arrow functions**: Incluir sempre parênteses

## Principais recursos

### Rigor do TypeScript

A configuração núcleo ativa as verificações estritas do TypeScript:

- Desencoraja o uso de `any` (`noExplicitAny: error`)
- Exige tratamento de `null/undefined`
- Prefere tipos explícitos em certas situações
- Impõe segurança de tipos em toda a base de código

### Equivalentes recomendados do ESLint

Melhores práticas comuns estão incluídas:

- Sem variáveis não utilizadas (`noUnusedVariables: error`)
- Sem imports não utilizados (`noUnusedImports: error`)
- Sem `eval` explícito (`noGlobalEval: error`)
- Sem poluição de protótipo (`noPrototypeBuiltins: error`)

### Acessibilidade por padrão

Regras equivalentes a padrões de acessibilidade:

- Atributos ARIA devem ser válidos
- Elementos HTML semânticos são preferidos
- Suporte à navegação por teclado é exigido
- Texto alternativo para imagens é obrigatório

### Suporte ao Node.js

Tratamento específico de ambiente:

- Reconhece globals do Node.js (`module`, `process`)
- Exige o protocolo `node:` para módulos internos
- Suporta tanto CommonJS quanto ESM

## Notas

Certas regras estão desativadas porque são lentas demais para rodar com o Scanner. Como indicado na [documentação do Biome](https://biomejs.dev/guides/investigate-slowness/#first-things-first), regras de projeto habilitadas são conhecidas por causar overhead de desempenho em troca de uma análise mais avançada.

Em nossos testes, descobrimos que essas regras podem fazer a diferença entre 300ms e 13s de lint.
