A configuração core do Ultracite fornece um conjunto abrangente de regras de linting independentes de framework e definições de formatação. Esta é a base que todas as outras configurações específicas de framework estendem.

## Instalação

A configuração core é incluída automaticamente quando você executa:

```bash
npx ultracite init
```

Ou adicione-a manualmente ao seu `biome.jsonc`:

```jsonc title="biome.jsonc"
{
  "extends": ["ultracite/core"]
}
```

## Visão geral

A configuração core inclui centenas de regras de linting rigorosas cobrindo:

- **Acessibilidade (a11y)**: Aplicar atributos ARIA, HTML semântico e navegação por teclado
- **Complexidade**: Reduzir complexidade cognitiva e aplicar melhores práticas
- **Correção**: Prevenir erros comuns e aplicar segurança de tipos
- **Performance**: Otimizar o código para melhor desempenho em tempo de execução
- **Segurança**: Prevenir vulnerabilidades de segurança
- **Estilo**: Aplicar estilo de código consistente
- **Suspeitos**: Detectar padrões suspeitos que podem indicar bugs

## Configurações do formatador

- **Indentation**: 2 espaços
- **Line Width**: 80 caracteres
- **Line Ending**: LF (estilo Unix)
- **Semicolons**: Sempre obrigatórios
- **Quotes**: Aspas duplas para JavaScript/TypeScript e JSON
- **Trailing Commas**: Estilo ES5 (arrays/objetos multilinha)
- **Arrow Parentheses**: Incluir sempre parênteses

## Principais recursos

### Rigor do TypeScript

A configuração core ativa as verificações estritas do TypeScript:

- Desencoraja o uso de `any` (`noExplicitAny: error`)
- Exige tratamento de `null/undefined`
- Prefere tipos explícitos em certas situações
- Aplica segurança de tipos em toda a base de código

### Equivalentes recomendados do ESLint

Regras comuns de melhores práticas estão incluídas:

- Sem variáveis não usadas (`noUnusedVariables: error`)
- Sem imports não usados (`noUnusedImports: error`)
- Sem eval explícito (`noGlobalEval: error`)
- Sem prototype pollution (`noPrototypeBuiltins: error`)

### Acessibilidade por padrão

Regras equivalentes a padrões de acessibilidade:

- Atributos ARIA devem ser válidos
- Elementos HTML semânticos preferidos
- Suporte a navegação por teclado requerido
- Texto alternativo para imagens requerido

### Suporte a Node.js

Tratamento específico por ambiente:

- Reconhece globais do Node.js (`module`, `process`)
- Obriga protocolo `node:` para módulos embutidos
- Suporta tanto CommonJS quanto ESM

## Notas

Certas regras estão desabilitadas porque são lentas demais para rodar com o Scanner. Como observado na [documentação do Biome](https://biomejs.dev/guides/investigate-slowness/#first-things-first), regras de projeto habilitadas são conhecidas por causar overhead de performance em troca de análises mais avançadas.

Em nossos testes, descobrimos que essas regras podem fazer a diferença entre um lint de 300ms e um de 13s.
