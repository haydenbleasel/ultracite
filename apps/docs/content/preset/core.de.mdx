Die Core-Konfiguration von Ultracite stellt eine umfassende Sammlung framework-agnostischer Linting-Regeln und Formatierungseinstellungen bereit. Dies ist die Grundlage, von der alle anderen framework-spezifischen Konfigurationen erweitert werden.

## Installation

Die Core-Konfiguration wird automatisch eingeschlossen, wenn Sie ausführen:

```bash
npx ultracite init
```

Oder fügen Sie sie manuell zu Ihrer `biome.jsonc` hinzu:

```jsonc title="biome.jsonc"
{
  "extends": ["ultracite/core"]
}
```

## Überblick

Die Core-Konfiguration umfasst Hunderte strenger Linting-Regeln, die folgende Bereiche abdecken:

- **Accessibility (a11y)**: Erzwingt ARIA-Attribute, semantisches HTML und Tastaturnavigation
- **Complexity**: Reduziert kognitive Komplexität und erzwingt Best Practices
- **Correctness**: Verhindert häufige Fehler und fördert Typsicherheit
- **Performance**: Optimiert Code für bessere Laufzeit-Performance
- **Security**: Verhindert Sicherheitslücken
- **Style**: Erzwingt konsistenten Code-Stil
- **Suspicious**: Erkennt verdächtige Muster, die auf Fehler hinweisen könnten

## Formatter-Einstellungen

- **Einrückung**: 2 Leerzeichen
- **Zeilenbreite**: 80 Zeichen
- **Zeilenende**: LF (Unix-Stil)
- **Semikolons**: Immer erforderlich
- **Anführungszeichen**: Doppelte Anführungszeichen für JavaScript/TypeScript und JSON
- **Trailing Commas**: ES5-Stil (mehrzeilige Arrays/Objekte)
- **Arrow Parentheses**: Klammern immer einschließen

## Hauptmerkmale

### TypeScript-Striktheit

Die Core-Konfiguration aktiviert TypeScripts strenge Prüfungen:

- Entmutigt die Verwendung von `any` (`noExplicitAny: error`)
- Erfordert die Behandlung von `null/undefined`
- Bevorzugt in bestimmten Situationen explizite Typen
- Erzwingt Typsicherheit im gesamten Codebasis

### Entsprechende empfohlene ESLint-Regeln

Häufige Best-Practice-Regeln sind enthalten:

- Keine ungenutzten Variablen (`noUnusedVariables: error`)
- Keine ungenutzten Imports (`noUnusedImports: error`)
- Kein explizites eval (`noGlobalEval: error`)
- Kein Prototype Pollution (`noPrototypeBuiltins: error`)

### Accessibility standardmäßig

Regeln, die Zugänglichkeitsstandards entsprechen:

- ARIA-Attribute müssen gültig sein
- Semantische HTML-Elemente werden bevorzugt
- Unterstützung für Tastaturnavigation erforderlich
- Alternativtext für Bilder erforderlich

### Node.js-Unterstützung

Umgebungsspezifische Behandlung:

- Erkennt Node.js-Globals (`module`, `process`)
- Erzwingt das `node:`-Protokoll für eingebaute Module
- Unterstützt sowohl CommonJS als auch ESM

## Hinweise

Bestimmte Regeln sind deaktiviert, weil sie zu langsam für den Scanner sind. Wie in den [Biome docs](https://biomejs.dev/guides/investigate-slowness/#first-things-first) beschrieben, sind projektweit aktivierte Regeln dafür bekannt, Leistungsaufwand zu verursachen, bieten dafür aber eine fortgeschrittenere Analyse.

In unseren Tests haben wir festgestellt, dass diese Regeln den Unterschied zwischen einem 300ms- und einem 13s-Lint ausmachen können.
