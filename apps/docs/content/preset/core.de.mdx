Die Core-Konfiguration von Ultracite bietet eine umfassende Sammlung framework‑agnostischer Linting‑Regeln und Formatierungseinstellungen. Dies ist die Grundlage, die alle anderen framework‑spezifischen Konfigurationen erweitern.

## Installation

Die Core-Konfiguration wird automatisch eingeschlossen, wenn Sie ausführen:

```bash
npx ultracite init
```

Oder fügen Sie sie manuell zu Ihrer `biome.jsonc` hinzu:

```jsonc title="biome.jsonc"
{
  "extends": ["ultracite/core"]
}
```

## Überblick

Die Core-Konfiguration enthält Hunderte strenger Linting‑Regeln, die Folgendes abdecken:

- **Zugänglichkeit (a11y)**: Erzwinge ARIA‑Attribute, semantisches HTML und Tastaturnavigation
- **Komplexität**: Reduziere kognitive Komplexität und erzwinge Best Practices
- **Korrektheit**: Verhindere häufige Fehler und erzwinge Typensicherheit
- **Performance**: Optimiere Code für bessere Laufzeit‑Performance
- **Sicherheit**: Vermeide Sicherheitslücken
- **Stil**: Erzwinge konsistenten Code‑Stil
- **Verdächtig**: Erkenne verdächtige Muster, die auf Bugs hinweisen könnten

## Formatierungseinstellungen

- **Einrückung**: 2 Leerzeichen
- **Zeilenbreite**: 80 Zeichen
- **Zeilenende**: LF (Unix‑Stil)
- **Semikolons**: Immer erforderlich
- **Anführungszeichen**: Doppelte Anführungszeichen für JavaScript/TypeScript und JSON
- **Trailing Commas**: ES5‑Stil (mehrzeilige Arrays/Objekte)
- **Klammern bei Arrow‑Funktionen**: Immer Klammern einschließen

## Hauptmerkmale

### TypeScript‑Strenge

Die Core‑Konfiguration aktiviert TypeScript‑strenge Prüfungen:

- Entmutigt die Verwendung von `any` (`noExplicitAny: error`)
- Erfordert die Behandlung von `null/undefined`
- Bevorzugt explizite Typen in bestimmten Situationen
- Erzwingt Typensicherheit über den gesamten Code‑Basis

### Empfohlene ESLint‑Äquivalente

Gängige Best‑Practice‑Regeln sind enthalten:

- Keine ungenutzten Variablen (`noUnusedVariables: error`)
- Keine ungenutzten Imports (`noUnusedImports: error`)
- Kein explizites eval (`noGlobalEval: error`)
- Kein Prototype‑Pollution (`noPrototypeBuiltins: error`)

### Barrierefreiheit standardmäßig

Regeln, die den Barrierefreiheitsstandards entsprechen:

- ARIA‑Attribute müssen gültig sein
- Semantische HTML‑Elemente werden bevorzugt
- Unterstützung für Tastaturnavigation erforderlich
- Alt‑Text für Bilder erforderlich

### Node.js‑Unterstützung

Umgebungsspezifische Handhabung:

- Erkennt Node.js‑Globals (`module`, `process`)
- Erzwingt das `node:`‑Protokoll für eingebaute Module
- Unterstützt sowohl CommonJS als auch ESM

## Hinweise

Bestimmte Regeln sind deaktiviert, weil sie mit dem Scanner zu langsam auszuführen sind. Wie in der [Biome‑Dokumentation](https://biomejs.dev/guides/investigate-slowness/#first-things-first) erwähnt, sind projektweit aktivierte Regeln dafür bekannt, Performance‑Overhead zugunsten einer fortgeschritteneren Analyse zu verursachen.

In unseren Tests haben wir festgestellt, dass diese Regeln den Unterschied zwischen ca. 300 ms und 13 s Lint‑Laufzeit ausmachen können.
