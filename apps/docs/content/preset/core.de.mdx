Die Kernkonfiguration von Ultracite bietet eine umfassende Sammlung framework-agnostischer Linting-Regeln und Formatierungseinstellungen. Dies ist die Grundlage, von der alle anderen framework-spezifischen Konfigurationen ausgehen.

## Installation

Die Kernkonfiguration wird automatisch eingeschlossen, wenn Sie ausführen:

```bash
npx ultracite init
```

Oder fügen Sie sie manuell zu Ihrer `biome.jsonc` hinzu:

```jsonc title="biome.jsonc"
{
  "extends": ["ultracite/core"]
}
```

## Übersicht

Die Kernkonfiguration enthält Hunderte strenger Linting-Regeln und deckt ab:

- **Barrierefreiheit (a11y)**: Erzwingt ARIA-Attribute, semantisches HTML und Tastaturnavigation
- **Komplexität**: Reduziert kognitive Komplexität und erzwingt Best Practices
- **Korrektheit**: Verhindert häufige Fehler und erzwingt Typensicherheit
- **Performance**: Optimiert Code für bessere Laufzeit-Performance
- **Sicherheit**: Verhindert Sicherheitslücken
- **Stil**: Erzwingt konsistenten Code-Stil
- **Verdächtig**: Erkennt verdächtige Muster, die auf Fehler hinweisen könnten

## Formatter-Einstellungen

- **Einrückung**: 2 Leerzeichen
- **Zeilenbreite**: 80 Zeichen
- **Zeilenende**: LF (Unix-Format)
- **Semikolons**: Immer erforderlich
- **Anführungszeichen**: Doppelte Anführungszeichen für JavaScript/TypeScript und JSON
- **Trailing Commas**: ES5-Stil (mehrzeilige Arrays/Objekte)
- **Pfeilklammern**: Immer Klammern einschließen

## Hauptmerkmale

### TypeScript-Striktheit

Die Kernkonfiguration aktiviert TypeScripts strenge Prüfungen:

- Entmutigt die Verwendung von `any` (`noExplicitAny: error`)
- Erfordert Umgang mit `null/undefined`
- Bevorzugt explizite Typen in bestimmten Situationen
- Erzwingt Typensicherheit im gesamten Codebase

### Empfohlene ESLint-Entsprechungen

Häufige Best-Practice-Regeln sind enthalten:

- Keine unbenutzten Variablen (`noUnusedVariables: error`)
- Keine unbenutzten Importe (`noUnusedImports: error`)
- Kein explizites eval (`noGlobalEval: error`)
- Kein Prototype-Pollution (`noPrototypeBuiltins: error`)

### Barrierefreiheit standardmäßig

Regeln entsprechend Barrierefreiheitsstandards:

- ARIA-Attribute müssen gültig sein
- Semantische HTML-Elemente werden bevorzugt
- Unterstützung für Tastaturnavigation erforderlich
- Alt-Text für Bilder erforderlich

### Node.js-Unterstützung

Umgebungsspezifische Handhabung:

- Erkennt Node.js-Globals (`module`, `process`)
- Erzwingt das `node:`-Protokoll für eingebaute Module
- Unterstützt sowohl CommonJS als auch ESM

## Hinweise

Bestimmte Regeln sind deaktiviert, weil sie zu langsam für den Scanner sind. Wie in den [Biome docs](https://biomejs.dev/guides/investigate-slowness/#first-things-first) erwähnt, verursachen aktivierte Projektregeln bekannten Performance-Overhead zugunsten einer fortgeschritteneren Analyse.

In unseren Tests fanden wir heraus, dass diese Regeln den Unterschied zwischen 300 ms und 13 s Linting-Zeit ausmachen können.
