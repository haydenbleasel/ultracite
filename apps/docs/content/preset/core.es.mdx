La configuración core de Ultracite proporciona un conjunto integral de reglas de linting y ajustes de formato independientes del framework. Esta es la base que extienden todas las demás configuraciones específicas de framework.

## Instalación

La configuración core se incluye automáticamente cuando ejecutas:

```bash
npx ultracite init
```

O agrégala manualmente a tu `biome.jsonc`:

```jsonc title="biome.jsonc"
{
  "extends": ["ultracite/core"]
}
```

## Resumen

La configuración core incluye cientos de reglas de linting estrictas que cubren:

- **Accesibilidad (a11y)**: Hacer cumplir atributos ARIA, HTML semántico y navegación por teclado
- **Complejidad**: Reducir la complejidad cognitiva y aplicar las mejores prácticas
- **Corrección**: Prevenir errores comunes y garantizar la seguridad de tipos
- **Rendimiento**: Optimizar el código para un mejor rendimiento en tiempo de ejecución
- **Seguridad**: Prevenir vulnerabilidades de seguridad
- **Estilo**: Hacer cumplir un estilo de código consistente
- **Sospechosos**: Detectar patrones sospechosos que podrían indicar errores

## Ajustes del formateador

- **Indentación**: 2 espacios
- **Ancho de línea**: 80 caracteres
- **Final de línea**: LF (estilo Unix)
- **Puntos y comas**: Siempre obligatorios
- **Comillas**: Comillas dobles para JavaScript/TypeScript y JSON
- **Comas finales**: Estilo ES5 (arrays/objetos multilínea)
- **Paréntesis en funciones flecha**: Incluir siempre paréntesis

## Características clave

### Comprobaciones estrictas de TypeScript

La configuración core activa las comprobaciones estrictas de TypeScript:

- Desalienta el uso de `any` (`noExplicitAny: error`)
- Requiere el manejo de `null/undefined`
- Prefiere tipos explícitos en ciertas situaciones
- Impone seguridad de tipos en toda la base de código

### Equivalentes recomendados de ESLint

Se incluyen reglas comunes de buenas prácticas:

- No variables sin usar (`noUnusedVariables: error`)
- No imports sin usar (`noUnusedImports: error`)
- No eval explícito (`noGlobalEval: error`)
- No prototype pollution (`noPrototypeBuiltins: error`)

### Accesibilidad por defecto

Reglas equivalentes a estándares de accesibilidad:

- Los atributos ARIA deben ser válidos
- Se prefieren elementos HTML semánticos
- Se requiere soporte de navegación por teclado
- Se requiere texto alternativo (alt) para imágenes

### Compatibilidad con Node.js

Manejo específico por entorno:

- Reconoce globals de Node.js (`module`, `process`)
- Exige el protocolo `node:` para módulos incorporados
- Soporta tanto CommonJS como ESM

## Notas

Ciertas reglas están deshabilitadas porque son demasiado lentas para ejecutarse con el Scanner. Como se indica en la [documentación de Biome](https://biomejs.dev/guides/investigate-slowness/#first-things-first), las reglas de proyecto habilitadas son conocidas por causar sobrecarga de rendimiento a cambio de un análisis más avanzado.

En nuestras pruebas, hemos encontrado que estas reglas pueden marcar la diferencia entre un lint de 300 ms y uno de 13 s.
