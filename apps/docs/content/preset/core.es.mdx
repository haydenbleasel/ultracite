La configuración core de Ultracite proporciona un conjunto exhaustivo de reglas de linting independientes del framework y ajustes de formato. Esta es la base que extienden todas las demás configuraciones específicas de framework.

## Instalación

La configuración core se incluye automáticamente cuando ejecutas:

```bash
npx ultracite init
```

O añádela manualmente a tu `biome.jsonc`:

```jsonc title="biome.jsonc"
{
  "extends": ["ultracite/core"]
}
```

## Descripción general

La configuración core incluye cientos de reglas estrictas de linting que cubren:

- **Accesibilidad (a11y)**: Aplicar atributos ARIA, HTML semántico y navegación por teclado
- **Complejidad**: Reducir la complejidad cognitiva y aplicar buenas prácticas
- **Corrección**: Prevenir errores comunes y aplicar seguridad de tipos
- **Rendimiento**: Optimizar el código para un mejor rendimiento en tiempo de ejecución
- **Seguridad**: Prevenir vulnerabilidades de seguridad
- **Estilo**: Aplicar un estilo de código consistente
- **Sospechosos**: Detectar patrones sospechosos que podrían indicar errores

## Ajustes del formateador

- **Indentación**: 2 espacios
- **Ancho de línea**: 80 caracteres
- **Fin de línea**: LF (estilo Unix)
- **Punto y coma**: Siempre requerido
- **Comillas**: Comillas dobles para JavaScript/TypeScript y JSON
- **Comas finales**: Estilo ES5 (arrays/objetos multilínea)
- **Paréntesis en arrow**: Incluir siempre paréntesis

## Características clave

### Estrictitud de TypeScript

La configuración core habilita las comprobaciones estrictas de TypeScript:

- Desalienta el uso de `any` (`noExplicitAny: error`)
- Requiere el manejo de `null/undefined`
- Prefiere tipos explícitos en ciertas situaciones
- Aplica seguridad de tipos en todo el código

### Equivalentes recomendados de ESLint

Se incluyen reglas comunes de buenas prácticas:

- No variables sin usar (`noUnusedVariables: error`)
- No imports sin usar (`noUnusedImports: error`)
- No `eval` explícito (`noGlobalEval: error`)
- No contaminación del prototipo (`noPrototypeBuiltins: error`)

### Accesibilidad por defecto

Reglas equivalentes a estándares de accesibilidad:

- Los atributos ARIA deben ser válidos
- Se prefieren elementos HTML semánticos
- Se requiere soporte de navegación por teclado
- Se requiere texto alternativo para imágenes

### Soporte para Node.js

Manejo específico por entorno:

- Reconoce globals de Node.js (`module`, `process`)
- Aplica el protocolo `node:` para módulos incorporados
- Soporta tanto CommonJS como ESM

## Notas

Ciertas reglas están deshabilitadas porque son demasiado lentas para ejecutarse con el Scanner. Como se indica en la [documentación de Biome](https://biomejs.dev/guides/investigate-slowness/#first-things-first), las reglas habilitadas por proyecto se sabe que pueden causar sobrecarga de rendimiento a cambio de un análisis más avanzado.

En nuestras pruebas, hemos encontrado que estas reglas pueden marcar la diferencia entre un lint de 300ms y uno de 13s.
