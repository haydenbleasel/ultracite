La configuración core de Ultracite proporciona un conjunto integral de reglas de linting y ajustes de formato independientes del framework. Esta es la base que todas las demás configuraciones específicas de framework extienden.

## Instalación

La configuración core se incluye automáticamente cuando ejecutas:

```bash
npx ultracite init
```

O añádela manualmente a tu `biome.jsonc`:

```jsonc title="biome.jsonc"
{
  "extends": ["ultracite/core"]
}
```

## Resumen

La configuración core incluye cientos de reglas de linting estrictas que abarcan:

- **Accesibilidad (a11y)**: Hacer cumplir atributos ARIA, HTML semántico y la navegación por teclado
- **Complejidad**: Reducir la complejidad cognitiva y aplicar buenas prácticas
- **Corrección**: Prevenir errores comunes y hacer cumplir la seguridad de tipos
- **Rendimiento**: Optimizar el código para un mejor rendimiento en tiempo de ejecución
- **Seguridad**: Prevenir vulnerabilidades de seguridad
- **Estilo**: Hacer cumplir un estilo de código consistente
- **Sospechoso**: Detectar patrones sospechosos que podrían indicar errores

## Ajustes del formateador

- **Indentación**: 2 espacios
- **Ancho de línea**: 80 caracteres
- **Final de línea**: LF (estilo Unix)
- **Punto y coma**: Siempre requerido
- **Comillas**: Comillas dobles para JavaScript/TypeScript y JSON
- **Comas finales**: Estilo ES5 (arrays/objetos multilínea)
- **Paréntesis en funciones flecha**: Incluir siempre paréntesis

## Características clave

### Rigor de TypeScript

La configuración core habilita las comprobaciones estrictas de TypeScript:

- Desalienta el uso de `any` (`noExplicitAny: error`)
- Requiere el manejo de `null/undefined`
- Prefiere tipos explícitos en ciertas situaciones
- Hace cumplir la seguridad de tipos en todo el código

### Equivalentes recomendados de ESLint

Se incluyen reglas comunes de buenas prácticas:

- No variables sin usar (`noUnusedVariables: error`)
- No importaciones sin usar (`noUnusedImports: error`)
- No uso explícito de eval (`noGlobalEval: error`)
- No contaminación de prototipos (`noPrototypeBuiltins: error`)

### Accesibilidad por defecto

Reglas equivalentes a estándares de accesibilidad:

- Los atributos ARIA deben ser válidos
- Preferencia por elementos HTML semánticos
- Requerir soporte de navegación por teclado
- Texto alternativo para imágenes requerido

### Soporte para Node.js

Manejo específico por entorno:

- Reconoce globals de Node.js (`module`, `process`)
- Hace cumplir el protocolo `node:` para módulos incorporados
- Soporta tanto CommonJS como ESM

## Notas

Ciertas reglas están deshabilitadas porque son demasiado lentas para ejecutarse con el Scanner. Como se indica en la [documentación de Biome](https://biomejs.dev/guides/investigate-slowness/#first-things-first), las reglas del proyecto habilitadas se sabe que causan sobrecarga de rendimiento a cambio de un análisis más avanzado.

En nuestras pruebas, hemos encontrado que estas reglas pueden marcar la diferencia entre un lint de 300ms y uno de 13s.
