Ultracite のコア構成は、フレームワークに依存しない包括的なリンティングルールとフォーマット設定のセットを提供します。これは、他のすべてのフレームワーク固有の構成が拡張する基盤です。

## インストール

コア構成は、次のコマンドを実行すると自動的に含まれます：

```bash
npx ultracite init
```

または手動で `biome.jsonc` に追加してください:

```jsonc title="biome.jsonc"
{
  "extends": ["ultracite/core"]
}
```

## 概要

コア構成には、数百に及ぶ厳格なリンティングルールが含まれており、以下をカバーします:

- **アクセシビリティ (a11y)**: ARIA 属性、セマンティック HTML、およびキーボード操作を強制
- **複雑さ**: 認知的複雑さを減らし、ベストプラクティスを強制
- **正確性**: 一般的なエラーを防ぎ、型の安全性を強制
- **パフォーマンス**: 実行時パフォーマンス向上のための最適化
- **セキュリティ**: セキュリティ脆弱性の防止
- **スタイル**: 一貫したコードスタイルを強制
- **疑わしいパターン**: バグを示唆する疑わしいパターンを検出

## フォーマッタ設定

- **インデント**: 2 スペース
- **行幅**: 80 文字
- **行末**: LF (Unix スタイル)
- **セミコロン**: 常に必要
- **引用符**: JavaScript/TypeScript と JSON には二重引用符を使用
- **末尾のカンマ**: ES5 スタイル（複数行の配列/オブジェクト）
- **アロー関数の括弧**: 常に括弧を含める

## 主要な機能

### TypeScript の厳格性

コア構成は TypeScript の厳密なチェックを有効にします:

- `any` の使用を抑制 (`noExplicitAny: error`)
- `null`/`undefined` の取り扱いを必須にする
- 特定の状況で明示的な型を優先
- コードベース全体で型安全性を強制

### ESLint 推奨の同等ルール

一般的なベストプラクティスのルールが含まれます:

- 未使用変数を禁止 (`noUnusedVariables: error`)
- 未使用の import を禁止 (`noUnusedImports: error`)
- 明示的な eval を禁止 (`noGlobalEval: error`)
- プロトタイプ汚染を禁止 (`noPrototypeBuiltins: error`)

### デフォルトでのアクセシビリティ

アクセシビリティ基準に相当するルール:

- ARIA 属性は有効でなければならない
- セマンティックな HTML 要素が推奨される
- キーボード操作のサポートが必須
- 画像には代替テキストが必要

### Node.js サポート

環境固有の取り扱い:

- Node.js のグローバル（`module`、`process`）を認識
- 組み込みモジュールに対して `node:` プロトコルを強制
- CommonJS と ESM の両方をサポート

| `useGuardForIn` | `error` | for-in ループに if 文のガードを含めることを要求します。 |
| `useIsArray` | `error` | `instanceof Array` の代わりに `Array.isArray()` を使用します。 |
| `useIterableCallbackReturn` | `error` | イテレートコールバックで一貫した戻り値を強制します。 |
| `useNamespaceKeyword` | `error` | TypeScript の namespace 宣言に module キーワードの代わりに namespace キーワードを使用することを要求します。 |
| `useNumberToFixedDigitsArgument` | `error` | `Number#toFixed()` に桁数引数を使用することを強制します。 |
| `useStaticResponseMethods` | `error` | 可能な場合に `new Response()` コンストラクタの代わりに静的な `Response` メソッドを使用します。 |
| `useStrictMode` | `error` | スクリプトファイルでディレクティブ `use strict` の使用を強制します。 |

## 注記

Scanner で実行すると遅すぎるために無効化されているルールがあります。詳細は [Biome のドキュメント](https://biomejs.dev/guides/investigate-slowness/#first-things-first) にあるように、有効にされたプロジェクトルールは、より高度な解析と引き換えにパフォーマンスオーバーヘッドを引き起こすことが知られています。

当社のテストでは、これらのルールが 300ms と 13s のリントの差を生むことがあることが判明しています。
