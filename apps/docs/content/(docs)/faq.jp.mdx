## Q: Ultraciteとは正確には何ですか — BiomeやESLintとはどう違いますか?

Ultracite は本質的に Biome 上に構築されたプリセット設定です。Ultracite はルールと設定をキュレーションしたバンドルと考えてください。一方で Biome は実際にコードをフォーマット／リントするエンジンです。Biome 単体でも使えますが、どのルールを有効にするかやオプションを手動で決める必要があります。Ultracite はエコシステム全体のベストプラクティスを使った、すぐに使える設定を提供することでその手間を省きます。

ESLint + Prettier と比較すると、Biome は Rust 実装とツールの統合（2〜3個のツールではなく1つのツール）のおかげで高速です。もし全ての ESLint ルールを細かく調整するのが好きな場合、Ultracite は制約が多く感じられるかもしれませんが、Ultracite の目的はその必要性を排除し、妥当なデフォルトを採用することです。

要するに: Biome はエンジン（リンター／フォーマッター）、Ultracite は設定です。

## Q: Ultracite を使うなら ESLint や Prettier はまだ必要ですか?

いいえ — Ultracite（経由で Biome）はあなたの Javascript／Typescript コードに対する ESLint と Prettier の機能を置き換えます。ESLint や Prettier を実行する必要はありません。実際、競合を避けるためにそれらの設定を取り除くことを推奨します。Ultracite の出力は通常、それらのツールが行っていたフォーマットとリンティングの大部分をカバーし、同等またはより厳格なルールを適用します。

唯一それらを使い続ける可能性があるのは、Biome がサポートしていない特殊なプラグイン（例えば非常に専門的な ESLint ルール）をどうしても使いたい場合です。しかし理想的には、Ultracite のエコシステムだけで作業することを想定しています。

## Q: JS/TS 以外のファイルはどうなりますか？Ultracite はそれらを処理しますか？

Ultracite は Biome を通じて、JavaScript、TypeScript、JSX、CSS、GraphQL に対してさまざまなレベルのリンティングとフォーマットサポートを提供します。

## Q: Ultracite はどのくらいの頻度で更新されますか？

Ultracite は必要に応じて更新され、しばしば Biome のリリースサイクルに追随します。Biome がまだ進化中であるため、Ultracite に頻繁なマイナーリリースがあるかもしれません。これらの更新は新しいルール、バグ修正、デフォルトの調整をもたらします。`-E` でピン留めされているため、あなたがアップグレードを選ばない限り予期しない更新を受けることはありません。

Ultracite のリポジトリのリリースを監視するか、[リリース](https://github.com/haydenbleasel/ultracite/releases) ページを確認するのが良いでしょう。Ultracite を更新する際はリリースノートを読んでください — ときどき新しいルールが有効になり、プロジェクトで新たな警告が出ることがあります（問題の検出には良いことですが注意が必要です）。Ultracite と一緒に Biome をアップグレードするのは、通常パッケージマネージャー経由で簡単です。

## Q: Ultracite が強制するルールに異議がある場合、どうすればいいですか？

いくつかの選択肢があります:

- `biome.jsonc` でそのルールをオフにするか、別のレベルに設定する（プロジェクトのための簡単な対処）。
- そのルールが全員に適用されるべきでないと思う場合は、GitHub リポジトリで [ディスカッション](https://github.com/haydenbleasel/ultracite/discussions/) を開く。メンテナがそれを厳しすぎると判断するか、オプションにできるかもしれません。
- スタイリング上の問題であれば、Ultracite の目的は設定よりも規約を優先することです — プロジェクト間で一貫性を保つためにツールのスタイルに合わせる価値がある場合があります。しかしもちろん、プロジェクトの要件が最優先です。

最終的には、プロジェクトのリント設定はあなたがコントロールします。Ultracite は出発点であり、自由にカスタマイズして構いませんが、理想的には小さな変更にとどめるべきです。多数のルールをオフにしているなら、Ultracite はあなたのチームの好みに合わない可能性があります（ただしそれは稀です）。

その場合、Biome の設定をゼロから構築することもできますが、多くの利便性を失うことになります。通常は少しの調整で Ultracite をうまく適合させられます。

## Q: VS Code / Cursor / Windsurf なしで Ultracite を使えますか？

はい — 私たちは VS Code とそのフォークに注力していますが（現代的なウェブ開発で最も人気のある IDE であるため）、Biome は CLI 経由や他の IDE と組み合わせて純粋に使用することができます。

Biome の [公式拡張](https://biomejs.dev/guides/editors/first-party-extensions/) と [サードパーティ拡張](https://biomejs.dev/guides/editors/third-party-extensions/) のエディタ統合をチェックして詳しく学んでください。

## Q: Ultracite が適用しているルールをどうやって知ればいいですか？一覧はありますか？

はい、GitHub リポジトリの [biome.jsonc](https://github.com/haydenbleasel/ultracite/blob/main/biome.jsonc) ファイルを確認すると、有効になっているルールが分かります。

## Q: Ultracite は見つけたすべての問題を自動修正できますか？

すべてではありませんが、多くは可能です。Ultracite は安全で決定的に修正可能な問題を自動修正します。これにはフォーマットの問題や多くのリントエラー（未使用の import を削除する、欠落している括弧を追加する、`==` を `===` に変更するなどのスタイル的または単純なコード変換）が含まれます。

より複雑な問題（ビジネスロジックや開発者の意図を必要とするもの）はあなた自身で修正する必要があります。例えば、命名規約に従って変数名を自動で置き換えたり、欠落しているエラーハンドリングコードを追加したりはしません — それらは警告として表示されます。

方針は ESLint と似ており、可能な限り修正を提供しますが、コードの振る舞いを変更してリスクを冒すことはしません。自動修正後に残った警告については、常に Problems パネルや CLI 出力を確認してください。

## Q: Ultracite が使っている Biome について詳しく知るにはどこを見ればいいですか？

[Biome](https://biomejs.dev/) には独自のウェブサイトとドキュメントがあり、そこでより技術的な詳細を見つけることができます。コードのフォーマット方法や特定の選択に関する哲学に興味があるなら、Biome のドキュメントが最良のリソースです。

加えて、Biome が Rome の後継として台頭した経緯に関するブログ記事やコミュニティ記事もあります。Biome を理解することで Ultracite の能力をよりよく理解できます。ただし、Ultracite を効果的に使うために Biome を深く学ぶ必要はありません — Ultracite の目的は多くのユーザーに対してそれらの詳細を抽象化することです。

## Q: 特定のプロジェクトやファイルで Ultracite を無効にするにはどうすればいいですか？

一時的に Ultracite を使いたくないプロジェクトがある場合、`biome.jsonc` に `extends: ["ultracite/core"]` を含めない（または設定ファイルを削除する）だけで済みます。VS Code 拡張がインストールされている場合、Biome のデフォルトで保存時にフォーマットを試みることがある点に注意してください。

完全に無効にするには、`formatOnSave` をオフにするか、そのワークスペース用に Biome 拡張をアンインストールすることができます。特定のファイルについては、先述の無視コメントやパターンを使用してください。基本的に Ultracite はプロジェクト単位でのオプトインです。設定されていなければ実行されません（ただし Biome 自体はトリガーされればフォーマットを行うことがありますが、リンティングルールは適用されません）。

## Q: インストール中に corepack エラーが出ます — どうすればいいですか？

一部のユーザーは、特に Node.js v22 上で古い corepack バージョンを使っている場合、`pnpm dlx ultracite init` を実行すると署名検証エラーに遭遇しています。エラーはパッケージインストール中に "Cannot find matching keyid" のように見えることがあります。

推奨される解決策は次の通りです:

- **corepack を更新する**: corepack をバージョン 0.34.0 以降にアップグレードしてください。`corepack prepare pnpm@latest --activate` を使うか、[pnpm の corepack セットアップガイド](https://pnpm.io/installation#using-corepack) に従ってください。
- **Node.js を更新する**: もし Node.js v20.x を使用している場合は、より新しい corepack を含む v22 以降へアップグレードを検討してください。
- **代わりに npx を使う**: corepack が問題を起こし続ける場合は、`pnpm dlx ultracite init` の代わりに `npx ultracite init` を使用できます。

これらの方法で解決しない場合は、Node.js と corepack のバージョン情報を添えて [Ultracite の GitHub リポジトリ](https://github.com/haydenbleasel/ultracite/issues) に issue を開いてください。
