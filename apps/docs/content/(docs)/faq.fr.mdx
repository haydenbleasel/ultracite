## Q : Qu'est-ce qu'Ultracite exactement – en quoi est-il différent de Biome ou ESLint ?

Ultracite est essentiellement une configuration prédéfinie construite sur Biome. Pensez à Ultracite comme un ensemble de règles et de paramètres sélectionnés, tandis que Biome est le moteur qui formate et analyse réellement le code. Vous pouvez utiliser Biome seul, mais vous devrez décider quelles règles activer et configurer ses options manuellement. Ultracite vous épargne cet effort en fournissant une configuration prête à l'emploi qui applique les meilleures pratiques de l'écosystème.

Comparé à ESLint + Prettier, Biome est plus rapide grâce à son implémentation en Rust et à son approche unifiée (un seul outil au lieu de deux ou trois). Si vous aimez déjà affiner chaque règle ESLint, Ultracite peut sembler contraignant ; toutefois, il vise à éliminer ce besoin en adoptant des valeurs par défaut sensées.

En bref : Biome est le moteur (linter / formatter), Ultracite est la configuration.

## Q : Ai-je encore besoin d'ESLint ou Prettier si j'utilise Ultracite ?

Non – Ultracite (via Biome) remplace la fonctionnalité d'ESLint et Prettier pour votre code JavaScript / TypeScript. Vous n'avez pas besoin d'exécuter ESLint ou Prettier du tout. En fait, il est recommandé de supprimer ces configurations pour éviter les conflits. La sortie d'Ultracite couvrira la plupart du formatage et du linting fournis par ces outils, généralement avec des règles équivalentes ou plus strictes.

Le seul cas où vous pourriez encore les utiliser est si vous avez un plugin très spécifique non pris en charge par Biome (par exemple, une règle ESLint très spécialisée). Idéalement, vous travailleriez entièrement avec l'écosystème Ultracite.

## Q : Qu'en est-il de mes fichiers non JS/TS ? Est-ce qu'Ultracite les gère ?

Ultracite, via Biome, offre différents niveaux de support de linting et de formatage pour JavaScript, TypeScript, JSX, CSS et GraphQL.

## Q : À quelle fréquence Ultracite est-il mis à jour ?

Ultracite est mis à jour selon les besoins, souvent en suivant le cycle de publication de Biome. Puisque Biome évolue encore, vous pouvez voir des versions mineures fréquentes d'Ultracite. Ces mises à jour peuvent apporter de nouvelles règles, des corrections de bugs ou des ajustements des valeurs par défaut. Étant donné qu'il est épinglé via `-E`, vous ne recevrez pas de mises à jour surprises à moins de choisir de mettre à niveau.

Il est conseillé de suivre le dépôt Ultracite pour les releases ou de consulter la [Releases](https://github.com/haydenbleasel/ultracite/releases) page. Lors de la mise à jour d'Ultracite, lisez les notes de version – il arrive parfois qu'une nouvelle règle soit activée et fasse apparaître de nouveaux avertissements dans votre projet (ce qui est positif pour détecter des problèmes, mais il faut en être conscient). Mettre à jour Biome en même temps qu'Ultracite se fait généralement de façon simple via le gestionnaire de paquets.

## Q : Si je ne suis pas d'accord avec une règle imposée par Ultracite, que dois-je faire ?

Vous avez plusieurs options :

- Configurez-la sur off ou à un autre niveau dans votre `biome.jsonc` (solution rapide pour votre projet).
- Ouvrez une [discussion](https://github.com/haydenbleasel/ultracite/discussions/) sur le dépôt GitHub si la règle devrait être ajustée pour tout le monde. Peut-être que les mainteneurs conviendront qu'elle est trop stricte ou pourrait être optionnelle.
- Si c'est une préférence stylistique, rappelez-vous que l'objectif d'Ultracite est la convention plutôt que la configuration – parfois il vaut mieux s'adapter au style de l'outil pour assurer la cohérence entre les projets. Mais bien sûr, les besoins de votre projet passent avant tout.

En fin de compte, vous contrôlez la config de lint de votre projet. Ultracite est un point de départ ; n'hésitez pas à le modeler, mais idéalement de façon mineure. Si vous vous retrouvez à désactiver la majorité des règles, alors Ultracite n'est peut‑être pas le preset adapté aux préférences de votre équipe (même si cela reste rare).

Dans ce cas, vous pouvez construire votre propre configuration Biome depuis zéro, mais vous perdrez beaucoup de commodité. En général, quelques ajustements suffisent pour que Ultracite convienne parfaitement.

## Q : Puis-je utiliser Ultracite sans VS Code / Cursor / Windsurf ?

Oui — nous nous concentrons sur VS Code et ses forks car ce sont les IDE les plus populaires pour le développement web moderne, mais Biome peut être utilisé uniquement via CLI ou avec d'autres IDE.

Consultez les intégrations éditeur [first-party](https://biomejs.dev/guides/editors/first-party-extensions/) et [third-party](https://biomejs.dev/guides/editors/third-party-extensions/) de Biome pour en savoir plus.

## Q : Comment savoir quelles règles Ultracite applique ? Y a-t-il une liste ?

Oui, consultez le fichier [biome.jsonc](https://github.com/haydenbleasel/ultracite/blob/main/biome.jsonc) dans le dépôt GitHub pour voir les règles activées.

## Q : Est-ce qu'Ultracite peut corriger tous les problèmes qu'il trouve ?

Pas tous, mais beaucoup. Ultracite corrigera automatiquement les problèmes qui sont sûrs et déterministes à corriger. Cela inclut les problèmes de formatage et de nombreuses erreurs de lint (particulièrement stylistiques ou des transformations simples du code comme la suppression d'importations inutilisées, l'ajout de parenthèses manquantes, le remplacement de `==` par `===`, etc.).

Les problèmes plus complexes (logique métier ou nécessitant une intention du développeur) sont laissés pour que vous les corrigiez. Par exemple, il ne renommerait pas automatiquement des variables pour suivre une convention de nommage ni n'ajouterait du code de gestion d'erreur manquant – il se contentera d'émettre un avertissement.

La philosophie est similaire à celle d'ESLint : fournir des corrections lorsque c'est possible, mais ne pas risquer d'altérer le comportement du code. Passez toujours en revue le panneau Problèmes ou la sortie CLI pour tout avertissement restant après l'auto-correction.

## Q : Où puis-je en apprendre davantage sur Biome (l'outil utilisé par Ultracite) ?

[Biome](https://biomejs.dev/) dispose de son propre site avec de la documentation, où vous trouverez plus de détails techniques. Si vous êtes curieux de savoir comment il formate le code ou la philosophie derrière certains choix, la documentation de Biome est la meilleure ressource.

De plus, il existe des billets de blog et des articles communautaires sur l'essor de Biome en tant que successeur de Rome. Comprendre Biome peut vous aider à mieux appréhender les capacités d'Ultracite. Cependant, vous n'avez pas besoin d'apprendre Biome en profondeur pour utiliser Ultracite efficacement – l'objectif d'Ultracite est d'abstraire ces détails pour la plupart des utilisateurs.

## Q : Comment désactiver Ultracite pour un projet ou un fichier spécifique ?

Si vous avez un projet où vous ne souhaitez temporairement pas Ultracite, vous pouvez simplement ne pas inclure `extends: ["ultracite/core"]` dans `biome.jsonc` (ou supprimer le fichier de configuration). Si l'extension VS Code est installée, elle peut quand même essayer de formater à la sauvegarde en utilisant les paramètres par défaut de Biome.

Pour désactiver complètement, vous pouvez désactiver `formatOnSave` ou désinstaller l'extension Biome pour cet espace de travail. Pour un fichier spécifique, comme mentionné, utilisez des commentaires d'ignorance ou des motifs d'ignorance. Essentiellement, Ultracite est activé par projet via la configuration. S'il n'est pas configuré, il ne s'exécutera pas (Biome lui‑même peut toujours exécuter le formatage si déclenché, mais pas les règles de linting).

## Q : J'obtiens une erreur corepack pendant l'installation – que dois-je faire ?

Certains utilisateurs ont rencontré des erreurs de vérification de signature en exécutant `pnpm dlx ultracite init` avec corepack, en particulier sur Node.js v22 avec des versions plus anciennes de corepack. L'erreur ressemble généralement à "Cannot find matching keyid" lors de l'installation d'un paquet.

Voici les solutions recommandées :

- **Mettre à jour corepack** : Passez à corepack version 0.34.0 ou ultérieure. Vous pouvez le faire avec `corepack prepare pnpm@latest --activate` ou en suivant le [pnpm corepack setup guide](https://pnpm.io/installation#using-corepack).
- **Mettre à jour Node.js** : Si vous êtes sur Node.js v20.x, envisagez de passer à v22 ou ultérieur, qui inclut une version plus récente de corepack.
- **Utiliser npx à la place** : Si corepack continue de poser problème, vous pouvez utiliser `npx ultracite init` au lieu de `pnpm dlx ultracite init`.

Si aucune de ces solutions ne fonctionne, ouvrez une issue sur le [Ultracite GitHub repo](https://github.com/haydenbleasel/ultracite/issues) en fournissant des détails sur vos versions de Node.js et corepack.
